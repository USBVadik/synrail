# DIRECT_LOOP_SPEC

## Goal

Establish the minimum direct operational loop for `Codex -> remote execution host -> execution agent CLI` so Codex can send one narrow task, receive literal output, validate proof, and decide the next safe step without human relay.

This spec is for the first direct loop only.
It does not change product code, Slice 2 semantics, or broader automation scope.

## Transport Path

- Local controller: `Codex`
- Remote host: `controller-defined target`
- Remote user: `controller-defined user`
- SSH auth method: `controller-defined dedicated automation key`
- Remote working directory: `controller-defined remote workdir`
- Execution agent entrypoint: `controller-defined CLI command`
- Transport mode: `ssh -> remote CLI -> raw output capture`
- Prompt handoff form: `headless prompt argument via --prompt`
- Response return form: `stdout with possible leading non-JSON prelude, followed by JSON response`

## Execution Contract

- Codex is the orchestrator and verifier.
- Gemini is the execution agent.
- One run = one narrow task.
- Gemini must stay within explicit scope.
- No hidden scope expansion.
- No success claim without proof bundle.
- Codex accepts or rejects based on evidence, not summary language.

For production-facing incident work, the standard loop now also requires:

- if a user/operator brings one narrow runtime clue, record it first as a bounded hypothesis
- do not treat that clue as diagnosis proof
- if the run claims target-surface truth, require target-surface attestation first
- if the run applies a production fix, require one post-deploy external runtime confirmation before accepting the fix claim

The relevant helpers are:

- `tools/reference/intake_incident_hypothesis.sh`
- `tools/reference/require_attested_target_surface.sh`
- `tools/reference/confirm_live_production_fix.sh`
- `tools/reference/incident_operator_flow.sh`

## Standard incident workflow

For a narrow production bugfix incident, the default workflow is now:

1. bounded incident intake
2. target-surface attestation
3. bounded probe or bounded patch review
4. exact patch attempt on the trusted surface
5. deploy/restart if the patch is intended for the live bot
6. one external runtime confirmation on the real scenario
7. closure or next-safe-step decision

Mandatory rules inside that workflow:

- step 1 may reduce search space, but may not count as proof
- step 2 is mandatory before any target-surface diagnosis/fix/test/deploy claim
- step 6 is mandatory before any production-fix claim is accepted

The smallest convenience wrapper for this workflow is now:

- `tools/reference/incident_operator_flow.sh start ...`
- `tools/reference/incident_operator_flow.sh confirm ...`

## Prompt Handoff

For each run, Codex sends:

- task id: `<RUN_ID>`
- mode: `<DESIGN | TOOLING | RECOVERY>`
- scope: `<EXACT_SCOPE>`
- allowed files/paths: `<ALLOWED_SCOPE>`
- forbidden files/paths: `<FORBIDDEN_SCOPE>`
- expected output format: `Response Contract from this file`
- proof requirements: `Proof Bundle from this file`

Prompt payload location:

- local source: `generated by Codex per run`
- remote destination: `passed inline to /usr/bin/gemini via --prompt`

Normalized unattended invocation baseline:

- `/usr/bin/gemini --prompt "<PROMPT>" --approval-mode plan --output-format json`

## Response Contract

Gemini response must contain these sections in order:

1. `Patch Attempt Status`
2. `Files Changed`
3. `Unified Diff`
4. `Read-Back Proof`
5. `Compile/Syntax Proof`
6. `Runtime/Scenario Proof` if relevant
7. `Artifact Identity`
8. `Next Safe Step`

If a section is not applicable, Gemini must say `NOT APPLICABLE` and why.

For the first direct probe, Gemini may return only the probe-specific sections requested by Codex, because the goal is transport validation rather than tooling execution.

## Proof Bundle

Minimum proof bundle for a tooling run:

- literal unified diff
- literal read-back of changed target areas
- compile or syntax output when relevant
- runtime/scenario output when relevant
- explicit statement of tested artifact
- explicit repo state after run

Minimum proof bundle for a read-only probe:

- literal command/output transcript
- explicit statement that no files changed
- explicit artifact identity
- raw stdout as returned by Gemini CLI, even if ANSI-formatted
- timeout/hang outcome if the probe does not complete cleanly
- if stdout begins with a non-JSON prelude, parsing begins at the first valid JSON object start

## Trusted Artifact Rules

- `working tree` must be named explicitly if Gemini touched uncommitted files.
- `commit artifact` must be named explicitly if Gemini tested a commit snapshot.
- `tested artifact` must be named explicitly in every run.
- For unattended operation, the authoritative artifact is the captured stdout of the headless run, with any leading non-JSON prelude excluded from JSON parsing.
- If artifact identity is ambiguous, the run is not trusted.
- If intended branch is not proven, the run is not accepted.
- Codex trusts repo files and literal outputs, not narrative summaries.

## Failure Modes

- SSH/auth failure
- wrong remote directory
- wrong branch or unknown branch
- Gemini CLI unavailable
- prompt handoff corruption
- truncated output
- ANSI/noisy stdout that prevents reliable read-back
- interactive confirmation blocking unattended completion
- stdin handoff attempt that never reaches response completion
- leading non-JSON prelude that breaks naive JSON parsing
- non-literal proof
- claimed patch that did not land
- wrong artifact tested
- hidden scope drift
- user/operator runtime clue ignored in favor of repeated broad rediscovery
- deploy completed without external runtime confirmation

Required reaction:

- stop normal flow
- classify failure
- request missing proof or switch to recovery

Additional required reaction for production bugfix work:

- if a narrow human runtime clue exists and is still relevant, record it as bounded hypothesis before more broad rediscovery
- if deploy/restart happened but no external runtime confirmation exists yet, do not accept the production-fix claim

Interactive block detection rule:

- treat the probe as blocked if no final model response is captured before timeout
- treat the probe as blocked if the CLI waits for confirmation or further input
- do not continue unattended after an interactive block is detected

## Minimal First Probe

Purpose: confirm transport, prompt handoff, raw response capture, and artifact clarity without changing product code.

Probe shape:

- mode: `DESIGN` or read-only `TOOLING`
- scope: inspect transport path and response identity only
- forbidden scope: any file modification
- required output:
  - Gemini echo of received task id
  - explicit statement of authoritative artifact identity
  - explicit statement that no files were modified
  - one short connectivity confirmation

Probe acceptance:

- transport succeeds
- raw output is captured intact
- artifact identity is explicit
- no file changes occurred
- the probe completes before timeout without interactive blocking
- headless invocation returns a usable JSON response after excluding any leading non-JSON prelude

## Required Connection Details

- `REMOTE_HOST_OR_IP` = `<controller-defined>`
- `REMOTE_SSH_USER` = `<controller-defined>`
- `SSH_KEY_PATH_OR_AGENT` = `<controller-defined>`
- `REMOTE_WORKDIR` = `<controller-defined>`
- `EXECUTION_AGENT_COMMAND` = `<controller-defined>`
- `PROMPT_HANDOFF_FORM` = `headless prompt argument via --prompt`
- `RESPONSE_RETURN_FORM` = `stdout with possible leading non-JSON prelude, followed by JSON response`
- `DEFAULT_REMOTE_BRANCH` = `<controller-defined>`
- `REMOTE_REPO_PATH` = `<controller-defined>`
- `ENV_SETUP_REQUIRED` = `NONE`
- `AUTHORITATIVE_ARTIFACT_AFTER_RUN` = `captured stdout of the probe run`

Remote invocation style:

- wrapper mode = `direct CLI`

Superseded unattended probe paths:

- raw stdin/heredoc probe path is superseded for unattended operation
- PTY/interactive UI probe path is superseded for unattended operation

## Open Questions

- Headless `--prompt` mode is now the proven unattended baseline.
- Raw response artifact for unattended runs is local capture of remote `stdout`.
- Gemini is called directly; no wrapper is currently required.
- Trusted remote baseline for product work remains unresolved because this spec does not establish product artifact trust.
- Output parsing must tolerate a leading non-JSON prelude such as `MCP issues detected...` before the JSON payload.
- Should the first probe run against the real repo checkout or a disposable copy?
