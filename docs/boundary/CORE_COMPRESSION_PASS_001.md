# Core Compression Pass 001

## Purpose

State the first bounded compression move after the initial baseline verdict.

This document exists so the project responds to value evidence by shrinking pressure, not by quietly growing around it.

## Compression trigger

The first baseline verdict showed:

- `Synrail better` on the killer path

But it also showed:

- clearly higher coordination overhead than the simpler baseline

That means the next product question is not:

- can we explain the system even better

It is:

- what can be compressed without losing the current advantage

## Keep

The following should remain clearly in the active core path:

1. target-surface attestation
2. exact-task identity discipline
3. proof-bearing artifact capture
4. proof-bundle acceptance/rejection
5. anti-false-success state reading

These are currently the strongest candidates for the minimum undeniable core.

## Compress

The following should now be treated as compression targets:

1. repo polish that does not improve comparison, reviewability, or closure economics
2. framing layers that restate value without changing decisions
3. helper breadth that is not required for the killer path
4. repeated meta-discussion that can be replaced by one record plus one decision rule

## Demote

The following should remain outside active core priority:

1. downstream capability policy
2. broad shell growth
3. public-facing expansion
4. new semantic layers not tied to baseline advantage

## Immediate next step

The next compression-aware move should be:

- keep future work tied to one of these:
  - stronger killer-path result
  - clearer baseline comparison
  - lower overhead for the same truth advantage

If a proposed change does not help one of those three, it should probably wait.

## What this pass does not do

This pass does not claim:

- the current core is final
- the current docs are already minimal
- the current helper family is already optimally small

It only sets the first disciplined shrink direction.

## Decision rule

After this pass, every new core-facing addition should answer:

- does this preserve the current advantage while lowering cost?

If not, the default should be to skip it for now.
