# Minimum Undeniable Core 001

## Purpose

Define the smallest core that still makes `Synrail` meaningfully better on the killer path.

This document exists to keep the project from confusing explanatory richness with necessary product weight.

## Core question

What is the minimum set of things that must exist for `Synrail` to beat the simpler baseline on the killer path?

Anything outside that set is not yet undeniable core.

## Current minimum undeniable core

The current minimum undeniable core appears to be:

1. target-surface attestation
2. explicit exact-task identity
3. proof-bearing artifact capture
4. proof-bundle based acceptance/rejection
5. blocker/closure state reading that rejects narrative-only success

These five together are the strongest current candidate for the smallest real product wedge.

## Why these belong

### 1. Target-surface attestation

Without it, execution-surface truth drifts and the rest of the path becomes unreliable.

### 2. Explicit exact-task identity

Without it, comparisons become vague and “similar enough” work hides failure.

### 3. Proof-bearing artifact capture

Without it, review depends on agent storytelling.

### 4. Proof-bundle acceptance/rejection

Without it, closure is still a narrative impression instead of a governed decision.

### 5. Anti-false-success state reading

Without it, the system cannot reliably tell “blocked,” “claimed,” and “accepted” apart.

## What is likely not undeniable core yet

The following may still be useful, but they are not yet proven undeniable:

- broader product-shell framing
- richer orientation docs
- future UI surfaces
- broad helper families
- anything aimed primarily at downstream capability policy

These should remain secondary until they clearly improve the killer path.

## Compression rule

If a component cannot answer:

- what breaks on the killer path if this is removed?

then it should not automatically remain in the active core.

## Next test

The next test for this minimum core is simple:

- compare the killer path against the simpler baseline
- observe which pieces of `Synrail` actually carried the advantage
- shrink or demote everything else

## Decision rule

Treat this document as provisional but binding enough for prioritization.

Until comparison disproves it, new core work should primarily strengthen one of the five elements above or improve how tightly they work together.
