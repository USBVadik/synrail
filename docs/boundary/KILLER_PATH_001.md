# Killer Path 001

## Purpose

Define the single most important adoption path for `Synrail` in its current phase.

This document exists to stop the project from proving many things weakly instead of proving one thing decisively.

## Killer path

The current killer path is:

- turn a noisy narrow exact-task bugfix attempt into an accepted closure-grade result without relying on narrative trust

In practical terms, this path is:

1. bounded exact task enters
2. target surface is attested
3. execution runs on an explicit baseline
4. artifacts are captured
5. proof bundle is reviewed
6. closure is either accepted or rejected honestly

## Why this path wins now

This is the best path to prove `Synrail` value because it is:

- painful in normal agentic work
- narrow enough to measure honestly
- strong enough to expose false success, dirty baselines, bad surface claims, and proof drift
- already supported by real proving-ground evidence

It is a better wedge than:

- broad orchestration
- repo polish by itself
- downstream capability tuning
- aspirational generality

## User-visible promise on this path

On this path, `Synrail` does not promise “better agents” in the abstract.

It promises something narrower and stronger:

- fewer false completions
- clearer blocker identity
- reviewable closure
- lower operator confusion about whether a task is actually done

## Narrow task class

The killer path currently stays inside:

- bounded router/trigger fixes
- small exact-task bugfixes
- controlled patch tasks

Do not widen beyond this class while this path is still the main proof surface.

## Success condition

The killer path is working only if it repeatedly produces:

- closure-grade exact-task results
- preserved artifacts
- acceptance or rejection through explicit proof rules

One elegant incident flow `PASS` is not enough.

One lucky exact success is not enough.

## Failure condition

The killer path is not yet working if:

- success is still mainly narrative
- proof bundles are still fragile
- baseline identity still drifts
- operator effort remains high while confidence remains low

## What this path excludes

This path does not currently try to prove:

- broad downstream capability value
- product-shell breadth
- benchmark generality
- public-release readiness

Those may matter later, but they are not the killer path.

## Immediate implication

Every near-term core change should answer one question:

- does this make the killer path more reliable, cheaper, or easier to review?

If not, it is probably not current-priority core work.
