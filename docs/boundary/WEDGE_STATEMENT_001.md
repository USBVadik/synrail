# Wedge Statement 001

## Purpose

State the narrowest product claim that current evidence actually supports.

This document exists so `Synrail` is sold to ourselves by its real wedge, not by a broader story than the evidence can carry.

## Current wedge

`Synrail` is most justified on:

- proof-sensitive closure paths
- false-success-heavy execution
- exact-task work where narrative confidence is not safe enough

More concretely:

- narrow exact-task bugfixes
- controlled patch tasks
- messy execution paths where wrong-surface claims, weak artifacts, or misleading success stories are plausible

## What Synrail is currently best at

Current strongest product claim:

- `Synrail` turns messy execution into reviewable closure more reliably than a simpler baseline on high-risk proof-sensitive paths

That is the current wedge.

Not:

- “better process everywhere”
- “better agents in general”
- “the right wrapper for every incident”

## Why this wedge is credible

This wedge is credible because:

- the strongest current comparison shows `Synrail better`
- the weaker comparison shows the baseline can still be good enough elsewhere
- therefore the evidence supports a narrow but real advantage, not a universal one

## User-facing translation

If we had to say it plainly:

- use `Synrail` when getting the truth wrong is expensive
- do not pay full `Synrail` cost when a simpler disciplined loop is already adequate

## Current application rule

Reach for the full governed path when the work has one or more of these properties:

- high false-success risk
- meaningful closure cost if accepted wrongly
- remote or ambiguous execution surface
- nontrivial artifact/proof ambiguity
- meaningful recovery cost after misleading output

If those are not present, the simpler baseline may already be enough.

## Decision rule

Any future product expansion should strengthen this wedge first.

If a proposal widens the product claim faster than it strengthens evidence on the wedge, it is probably premature.
