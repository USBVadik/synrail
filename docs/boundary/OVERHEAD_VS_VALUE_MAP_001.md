# Overhead vs Value Map 001

## Purpose

Map which current `Synrail` elements appear to create real value on the killer path and which elements currently look more like overhead.

This document exists so the next compression pass is grounded in comparison evidence rather than taste.

## Reading rule

This map is provisional.

It is not a permanent ontology.

It is a current reading from:

- the killer path
- the first baseline comparison
- the current repeatability evidence

## Strong value carriers

These currently look like real value carriers on the killer path:

### 1. Target-surface attestation

Why it matters:

- it prevents wrong-surface confidence
- it stops fake progress on the wrong host/repo path

Current reading:

- high value
- acceptable overhead

### 2. Exact-task identity discipline

Why it matters:

- it prevents “similar enough” drift
- it keeps comparison and closure review honest

Current reading:

- high value
- low to moderate overhead

### 3. Proof-bearing artifact capture

Why it matters:

- it defeats narrative-only success
- it makes closure reviewable

Current reading:

- very high value
- acceptable overhead

### 4. Proof-bundle acceptance/rejection

Why it matters:

- it converts messy execution into explicit closure decisions
- it is one of the clearest advantages over the simpler baseline

Current reading:

- very high value
- moderate overhead

### 5. Anti-false-success state reading

Why it matters:

- it sharply reduced false completion exposure on the killer path
- it made repeated machine/narrative contradiction survivable instead of corrosive

Current reading:

- very high value
- acceptable overhead

## Mixed-value components

These currently look useful, but still carry meaningful cost pressure:

### 1. Doctor/evaluation layering

Why it matters:

- it improves explicit state reading
- but it may currently be heavier than strictly necessary for every use

Current reading:

- real value
- moderate to high framing overhead

### 2. Reference helper set as a family

Why it matters:

- some helpers clearly support the killer path
- but the family can still look broader than the minimum wedge

Current reading:

- mixed value
- moderate overhead

### 3. Repeatability framing

Why it matters:

- repeatability matters
- but repeated discussion about it can become more expensive than the next proof step

Current reading:

- real value
- easy to over-extend

## Current overhead-heavy zones

These currently look more overhead-heavy than value-proven:

### 1. Broad repo-native polish beyond review utility

Current reading:

- low direct killer-path value
- easy scope creep risk

### 2. Surfaces that mainly improve explanation rather than decisions

Current reading:

- some value for readability
- weak value if they do not change closure economics

### 3. Anything aimed at downstream capability policy

Current reading:

- out of core scope for this phase
- should not consume active core budget

## Compression implication

The next compression pass should:

- preserve the strong value carriers
- simplify mixed-value layers where possible
- demote overhead-heavy zones out of the active core path

## Decision rule

If a component has:

- high explanatory weight
- low visible impact on false-success reduction, blocker truth, or reviewable closure

then it should be treated as compression pressure, not default core growth.
