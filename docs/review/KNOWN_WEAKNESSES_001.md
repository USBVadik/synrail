# Known Weaknesses 001

This document is intentionally blunt.

These are the weak spots we most want attacked.

## 1. Semantic proof is stronger than before, but still not domain-deep

The kernel now distinguishes structural completeness from semantic sufficiency.

That is a real improvement.

But it still does not prove arbitrary domain correctness.

What to attack:

- cases where the bundle is semantically sufficient by current rules but still too weak for the actual task
- cases where readback/scenario evidence can look “substantive enough” without really proving the change

## 2. Acceptance truth is more independent, but still runtime-generated

Acceptance criteria are no longer silently rebuilt during `check`, and they now require explicit refresh.

That is good.

But they are still generated by the same system family that consumes them.

What to attack:

- whether the criteria source is independent enough to deserve trust
- whether refresh semantics are too easy or too opaque
- whether criteria drift can still be masked in odd contours

## 3. Doctor coverage is measured, but the corpus is still curated

Doctor is no longer just waving at a declared profile.

But the measured part is still bounded by a selected fail-mode corpus.

What to attack:

- fail modes that should be critical but are not yet represented
- false confidence from a narrow but green corpus
- readiness cases where doctor should say “I do not know” more often than it currently does

## 4. Continuation arbitration is executable, but still narrow

There is now a real arbiter and real conflict trace.

That is a meaningful hardening step.

But it still focuses on the critical surfaces we selected.

What to attack:

- conflict sets that remain under-resolved
- cases where author intuition is still doing hidden work
- cases where a second operator would still not understand why the chosen source won

## 5. The alpha shell is usable, but still not obviously cheap

The current shell is much better than the earlier artifact-heavy path.

Still, the product can look like justified ceremony instead of obvious workflow leverage.

What to attack:

- any step that feels like translation of internal artifacts instead of real workflow help
- whether `save`, `repair-step`, `retry`, and telemetry export are actually the right outer verbs
- where the shell still leaks compatibility-era assumptions

## 6. The current strongest proof is still narrow alpha proof, not broad product proof

This repository now proves one stronger contour.

It does not prove broad product inevitability.

What to attack:

- whether the current wedge is strong enough to deserve expansion
- whether the product beats simpler substitutes often enough
- whether this should remain a narrow kernel instead of growing into a bigger shell
